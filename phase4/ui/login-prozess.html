Wie funktioniert der Login-Prozess?

  1. Du gibst im Feld User einen bekannten Benutzernamen ein.
  2. Wenn der User in der User-Liste existiert, wird die Rolle automatisch vorbelegt.
  3. Mit Login wird die Session lokal gespeichert (Browser storage).
  4. Bei API-Aufrufen sendet die UI die Header X-User und X-Role.
  5. Das Backend prueft die Identitaet gegen die Access-Policy / User-Registry.
  6. Nur erlaubte Projekte und Rollen duerfen Such- oder Schreibaktionen ausfuehren.

Was wird pro User gespeichert?

  • Eigene Chat-Sessions (inkl. Chat-Namen, Verlauf, Kontextdateien).
  • Aktiver Workspace und projektspezifische Voreinstellungen.
  • Die letzte Session, damit ein Reload direkt dort weitermacht.
Rollenwirkung:
admin/service: duerfen Write Actions wie Re-Index ausfuehren.
reviewer: read-only, keine schreibenden Aktionen.

Fehlerbilder schnell verstehen

  • 401 missing user identity: Kein Login oder leerer Username.
  • 403 user not allowed: User nicht in Policy/Registry.
  • 403 role mismatch: Rolle in UI passt nicht zur hinterlegten Rolle.
  • 403 project access denied: User darf das gewaehlte Projekt nicht sehen.
